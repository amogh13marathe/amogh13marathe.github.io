Why I built this

The problem API PMs don't talk about

In my work scaling SquareTrade's data ingestion platform to 80+ global retail partners, the most consistent friction point wasn't the API design itself — it was the documentation. Partners would submit incorrect payloads, misinterpret optional vs. required fields, and hit undocumented error states. Every gap in the docs became a support ticket or a delayed integration.

The industry has no consistent standard for what "good" API documentation looks like. Stripe is widely regarded as the gold bar, but no PM tooling exists to objectively measure the gap between any given doc and that bar — or to generate a prioritized fix list that maps directly to sprint work.

I built the API DX Scorecard to solve that. It's a tool I'd want to run against my own API docs before shipping, and against a competitor's before a competitive review.

The rubric

Six dimensions, weighted by developer impact

The scoring rubric is based on what drives time-to-first-call — the single best predictor of developer activation. Weights are set by what matters most to a developer trying to integrate:

Clarity
20%
Can a developer understand what this endpoint does within 30 seconds of landing?
Completeness
20%
Every field, type, default, and response shape — no guessing required.
Example Quality
20%
Copy, paste, get a successful response on the first try.
Error Handling
15%
What can go wrong, how errors are shaped, and how to recover.
Onboarding Friction
15%
Minutes to first successful authenticated call for a brand-new developer.
Consistency
10%
HTTP verbs, naming conventions, pagination — follows the patterns a developer expects.
Technical design

Fully free stack, production-quality output

A deliberate constraint: build it on tools any PM or startup can afford. No paid APIs, no proprietary infra. The scoring model needed to be strong enough to catch real documentation gaps, not produce generic feedback.

Gradio 5 — UI framework Groq — LLM inference API (free tier) Llama 3.3 70B — scoring model BeautifulSoup — URL doc scraping HuggingFace Spaces — hosting ($0) Python — everything else

The system prompt instructs the model to behave as a Senior Staff PM for Developer Experience — evaluating docs with evidence, not opinion. JSON-mode output enforces structured scores, which then render as a visual HTML scorecard with color-coded bars, letter grades, and export formats for Jira and GitHub Issues.

Key product decisions

What I prioritized and why

  • Evidence-based scores over vibes. Each dimension requires the model to quote or paraphrase the actual documentation. This prevents hallucinated praise and makes the output directly actionable.
  • Prioritized fix list, not a bug report. The output is structured as P0/P1/P2 improvements with "why it matters" reasoning — ready to drop into a sprint without editing.
  • Side-by-side comparison mode. The most common use case for competitive review is comparing your docs against a competitor's. Built as a native tab, not an afterthought.
  • Jira + GitHub export. The scorecard generates properly formatted Jira wiki markup and GitHub issue markdown with collapsible evidence sections — PM to engineer handoff in one click.
  • Don't grade on a curve. The system prompt explicitly tells the model that most public API docs score 5–7, and Stripe-tier is rare. This prevents the LLM's default tendency toward flattery.
Relevance to my PM work

What this demonstrates

This tool reflects how I think about developer experience as a PM discipline: the documentation is the product interface for developers, and it deserves the same rigor as a UI design review. At SquareTrade, improving API docs and the testing sandbox was one of the highest-leverage investments in the partner onboarding pipeline — it drove a 40% reduction in integration time without touching the API itself.

Building this also required the same skills I apply in production: designing a scoring rubric with defensible weightings, writing a structured system prompt that produces consistent output, and deciding what format makes the output most useful to the people who receive it (a PM handing off to an engineer).
